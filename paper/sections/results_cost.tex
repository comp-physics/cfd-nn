\section{Computational Cost Analysis}
\label{sec:cost}

We now turn to the central question of this paper: what does it cost to run each turbulence closure in a GPU-accelerated solver?
All models are implemented in the same solver, use the same spatial discretization, time integration, and Poisson solver, and are timed on the same hardware.
This eliminates confounding factors that arise when comparing models across different codes.

Several important conclusions emerge:

\paragraph{TBNN inference dominates its own cost.}
The NN forward pass (matrix-vector multiplies with $\tanh$ activations) accounts for 78\% of the TBNN's turbulence update time.
Feature computation (velocity gradients, invariants, tensor basis) is comparatively negligible at 0.5\%.
This means that reducing the network size (e.g., fewer layers, smaller hidden dimensions) would translate nearly directly to proportional cost reduction.

\paragraph{TBRF is bottlenecked by GPU$\to$CPU synchronization.}
The feature computation phase for TBRF requires 10.9~ms---the same invariant and tensor basis computation that takes only 0.5~ms for the fully GPU-resident TBNN.
The $22\times$ slowdown is entirely due to the GPU$\to$CPU memory transfer required to run the tree traversal on the CPU.
At $128^3$ (2.1M cells), this synchronization transfers approximately 2.1M $\times$ 5 features $\times$ 8 bytes $= 84$~MB plus 2.1M $\times$ 10 bases $\times$ 6 components $\times$ 8 bytes $= 1.0$~GB of tensor basis data from GPU to CPU.

\paragraph{Tree inference scales linearly with tree count.}
The tree traversal itself is fast on CPU: 2.6~ms for 1~tree vs.\ 39.3~ms for 10~trees ($15\times$ for $10\times$ more trees), showing near-linear scaling.
This confirms that the tree traversal is compute-bound on CPU, not memory-bound.
A fully GPU-resident tree implementation would eliminate the 10.9~ms synchronization overhead but introduce warp divergence; the net effect on performance is an open question.

\subsection{Why TBNN costs $20\times$ more than the base solver}

The TBNN's 98.4~ms turbulence update on a 4.84~ms base step is the most extreme overhead of any model tested.
Three factors contribute:

\begin{enumerate}
    \item \textbf{High per-cell FLOP count}: The TBNN performs $\sim$10{,}300~FLOPs per cell for the NN forward pass, invariant computation, and tensor basis contraction.
    The base solver's stencil operations (convection + diffusion) require only $\sim$100 FLOPs per cell per field component, and the FFT Poisson solver does $O(\log N)$ work per cell.
    The TBNN's per-cell cost exceeds the entire base solver step by roughly $100\times$.

    \item \textbf{Layer-sequential execution}: The 4-layer TBNN requires 4 serial matrix-vector products.
    Within each layer, the parallelism is over (cell, neuron) pairs: at $128^3$ with 64 neurons, this yields 134M parallel work items---sufficient to saturate the H100---but the serial layer dependency means the GPU completes one layer before starting the next.

    \item \textbf{Activation cost}: The TBNN evaluates 192~$\tanh$ functions per cell (64 per hidden layer $\times$ 3 layers).
    On GPU, $\tanh$ takes $\sim$$20$ cycles versus $\sim$$4$ for a multiply, so the 192 activations cost roughly $192 \times 20 = 3{,}840$ effective cycles per cell---comparable to the matrix-vector products themselves.
\end{enumerate}

\subsection{TBRF cost anatomy}
\label{sec:tbrf_cost}

The TBRF's cost structure differs fundamentally from the TBNN because its inference engine (decision tree traversal) is not GPU-resident in the current implementation.
The full execution pipeline per time step is:

\begin{enumerate}
    \item \textbf{Feature computation on GPU} ($< 1$~ms): Compute velocity gradients, strain/rotation rate tensors, non-dimensionalize by $k/\varepsilon$, compute 5 Pope invariants and 10 tensor basis tensors.
    This uses the same GPU kernels as the TBNN.

    \item \textbf{GPU$\to$CPU transfer} ($\sim$10~ms): Copy the invariant and tensor basis arrays from GPU to CPU memory.
    At $128^3$, this transfers $\sim$1.1~GB of data across the PCIe/NVLink bus.

    \item \textbf{Tree traversal on CPU} (2.6--39.3~ms): For each cell, for each of 10 basis coefficients, traverse all trees and average.
    Each tree traversal is a depth-$\sim$20 sequence of comparison-and-branch operations.
    The tree data structure (56~MB for 10 trees) fits in CPU L3 cache.

    \item \textbf{Tensor contraction + CPU$\to$GPU transfer}: Reconstruct $b_{ij} = \sum_n g^{(n)} T^{(n)}_{ij}$ and copy results back to GPU.
\end{enumerate}

The GPU$\to$CPU synchronization in step~2 accounts for 28\% of the TBRF-1 total cost and 14\% of TBRF-10.
A fully GPU-resident implementation would eliminate this overhead but face the fundamental challenge of warp divergence: cells within the same warp follow different paths through each tree, serializing execution within each 32-thread warp.
Whether the elimination of data transfer overhead outweighs the warp divergence penalty is an open question that we leave for future work.
